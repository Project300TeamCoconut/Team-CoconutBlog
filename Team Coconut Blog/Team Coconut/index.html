Introduction

Posted by Kian , internet

We are a group of 3rd year computing students in IT SLigo that have formed the group known as Team Coconut, and we intend to develop a movie application using swipable cards. We are using this blog to document all of our progress along the way so people can get an understanding of what buildng an andriod app from scratch. Our idea came from when you and a friend or a partner are at home and you can't decide what to watch, well you then simply use our app, pick a genre and our app will the list movies for you to swipe either to left for no or to the right for yes and at the end it will show you and your partner each movie that you both swiped right to.

Posted by Team Coconut | Filed under , internet

October

Our next task was to populate the cards with movie names and images from the API. We decided to just use movie names to make it simple until the app was up and running. We would then move onto using the posters and ratings. We had previously created a sample app to get the API working in android studio. It displayed the data from the API in a recycler view. This a list of movies with their name, rating and image. The recycler view is a library for android studio which only displays what the user can see on the screen, anything else is executed and not seen. Transferring this logic of the recycler view onto the viewholder We created a customer array adapter for the API but this used the recycler view so it could not be used within the cards. Instead we created another customer adapter “to provide views for an AdapterView, Returns a view for each object in a collection of data objects you provide, and can be used with list-based user interface widgets such as ListView or Spinner”. We then needed to populate the view on the cards by calling each object such as the movie name or image and place the result into a TextView or ImageView. Eventually through trial and error we got this array adapter to work by overriding the getView method to populate the cards view and its resource.

November

Our task this month was to register the likes and dislikes to firebase

Registering the likes and dislikes to firebase When the cards were populated with the data from API we needed to register swipes to the database to that specific user. We followed a tutorial on referencing and adding root nodes to the firebase database. We learned that it was quite simple, you must create a database reference first. Firebase provide a function to do this which was easy to access as we had already implemented the firebase reference in the build gradle. From here we could then create a reference in the OnLeftCardExit and OnRightCardExit methods which will add the movie name into a yes or no child in the database depending on the direction of the swipe within the current user ID that is logged into the app at that moment.

December

We have finally got the matches to display

The final step in having all the main functionality working was to display the matches for the current user. To do this we had to create a match’s activity with an xml file to get the matches and put them into an array list in order to be displayed. We then had to create a match’s model object which contained a constructor for the movie name, which is what we are retrieving to be displayed. Finally, we created a custom matches adapter to display the matches to the user using the xml file and a recycler view. There were two methods in the matches adapter that we needed to use in order for the display to work. The adapter extended a recycler view, similar to what we had used in the sample API project. onCreateViewHolder which is used to initialize the ViewHolder “It is the very object that represents each item in our collection and will be used to display it” (WillowTree Apps, 2016) and onBindViewHolder which is called when we need to pass the data into the ViewHolder. Basically, the adapter that contains these two methods prepares the xml file for the list of movie matches to be displayed. The actual fetching of the data takes place in the matches activity. Finally, in the matches activity we created a database reference pointing to the matches child node and notified the matches adapter of any changes such as, the user views their matches, comes back and swipes on more movies and matches with another few, views the matches again and they are updated with the new matches. we also did our poster

January

User Interface

We then started the UI design by creating wireframes in Mockflow. These sample interfaces helped us later on with the actual implementation. A copy of our wireframes can be seen in Appendix A. We wanted the UI to be simple and clean. We also did not want to bombard the user with many different sections or navigation bars. We added a simple background along with a logo, changed the font style and shape of the buttons. In regards to the actual cards themselves, we positioned them towards the middle of the screen so the poster and movie name can be easily seen. Although the matches and the watch list do not contain the poster, they show the movie name. There is also a delete button located beside each name.

March

Finalisng the application

We are testing the app now to make sure it is ready and we have shownit to some family and friends. The friends and family that we have shown the app to have all responded with positive feedback. No one really ever wants to decide what to watch or pick for someone else. We demoed the app and allowed users to interact and they all understood what we were trying to achieve. The tinder swipe effect has come into play here because it is familiar to most people and they understand how the swiping works. The users were able to view their matches or add to the watchlist. They could also clear all their matches and swipes to start from the beginning. Overall we had a great response from potential users which gave us important feedback, such as whether or not people understood the app, if they liked the UI or found it too complicated.
